## 设计原则（写进 ADR 作为 downstream 参考）

凡是面向用户的字符串，**必须杜绝以下元素**：
1. 内部决策编号（`Q3=c` / `PRD §14` / `Track A`）
2. 未来时间承诺（`Q3`、`推迟`、"未来接入"）—— 用"示例模式 / 示例视图"代替，不承诺时间
3. 文件路径 / 代码符号（见批 A BUG-23 / BUG-24）
4. 撒谎式夸大（本未接入却写"已接入"）
5. 实现层术语（`mock` 作为独立单词）

代码内注释（`//` / `/** */`）**可以保留**以上词汇，帮助下一位工程师理解 roadmap。批 C 只改"用户能看到的字符串"。

## Consequences

**正面**
- `/mcp` 页面去 roadmap 化，但仍然告诉用户"这里是示例"，不欺骗
- 资产详情 RAGFlow 摘要不再自称"已接入"（用户点过来肯定能分辨，暴露内部不一致是信任损耗）
- 代码内部注释留存 `Q3=c` 等 roadmap 标签供开发追溯

**负面 / 取舍**
- "示例模式 / 示例视图" 比 "mock" 对工程师略不直观；但工程师能从代码注释看到真实状态，权衡可接受
- 没有一次性 audit 整站其他可能的 Dev 文案（治理页、QA、Agent）。未来若测试再报，再做一轮扫描
