## Decision · 6 条修复

| Bug | 位置 | 根因 | 修复 |
|---|---|---|---|
| BUG-05 IAM "权限规则" 标签点击后 5 秒+"加载中…" 不消失 | `apps/web/src/knowledge/Iam/RulesTab.tsx::load()` | `listRules()` reject / timeout 时只 setErr，`rules` 仍为 null，UI 永久停在 "加载中…"（同时显示 error + spinner） | catch 分支追加 `setRules([])`，让 UI 脱离"加载中…"态并展示 error |
| BUG-06 多页面 Tab 点中后文字消失（只剩背景色） | `apps/web/src/index.css` `.kc-subtab:hover` 在 `.kc-subtab.active` **之后**定义 | `.kc-subtab.active` 是 `background: var(--p); color: #fff`；`.kc-subtab:hover` 把 color 覆盖为 `var(--p)` → **已激活 tab 悬停时紫字紫底，文字视觉消失** | `.kc-subtab:not(.active):hover` 排除激活项；`.kc-subtab.active:hover` 显式维持白字紫底；`.kc-tab` 同步加 `.kc-tab.active:hover` 防御 |
| BUG-08 搜索提交后结果区仍显示 "输入关键词开始搜索" | `apps/web/src/knowledge/Search/index.tsx` | 用 `useDebounce(query, 300)`；**debounce 等待期**（0–300ms）`loading=false`，空 state 继续渲染；用户以为"点击无反应" | 计算 `isTyping = query.trim().length≥2 && query !== debouncedQuery`；loading OR isTyping 都视为 "搜索中…" 状态 |
| BUG-11 新建空间/改密/新建用户 3 处空提交静默 | `SpaceTree/CreateSourceModal.tsx` · `auth/ChangePasswordModal.tsx` · `Iam/UsersTab.tsx::CreateUserModal` | 按钮 `disabled={busy \|\| !X \|\| Y.length < N}` 在空态时灰着，用户点击无反应 | 按钮 disabled 缩回只剩 `busy`；把校验下沉到 onClick handler 里 `setErr('...')` + return；ChangePasswordModal + CreateUserModal 补齐缺失的空态文案 |
| BUG-12 /ingest 网页抓取 URL 空点"抓取"无反馈 | `Ingest/FetchUrlTab.tsx::submit()` | 同上，按钮 `disabled={busy \|\| !url.trim()}` 吞反馈 | 校验改成 `if (!url.trim()) { setError('请输入 URL'); return }`；按钮 disabled 只保留 `busy` |
| BUG-13 批量任务"上传"按钮文案与提示矛盾 | `Ingest/ZipImporter.tsx::handleUpload()` + 按钮 | 按钮 `disabled={!file \|\| !bookId \|\| uploading}`；右侧空态文案说"请在左侧上传 ZIP"，但用户想点的按钮在左侧且是灰的 | 拆校验到 handleUpload：缺文件 / 缺知识库分别 setErr；按钮 disabled 只保留 `uploading`；加 `title` 属性做鼠标悬停 tooltip |

## 设计选择：为什么统一"去 disabled + onClick setErr"

报告里 3/6 条 bug 都是同一种：**按钮 disabled 把"为什么不能点"藏起来**。可选方案比较：

| 方案 | 优 | 缺 |
|---|---|---|
| A · 去 disabled（只留 busy），点击时 setErr | 用户点击立即看到"请输入 XX"；心智负担低 | 按钮永远可点，看起来"没有禁用"反馈 |
| B · 保留 disabled，旁边加 hint 文案 | 保留原禁用视觉 | 多一段布局；hint 文字多处重复 |
| C · HTML5 required + 浏览器 bubble | 零代码 | 样式不可控，和整站设计语言冲突 |

选 **A** —— 和报告里用户的期望最一致（"我点了，告诉我为什么不行"）。B/C 留给下一轮 UX 改造，若需要全局"禁用+原因"组件再统一做一次。

## CSS 锁定

新的 Tab 状态表（`index.css`）：

```css
/* kc-subtab：未激活 → muted 底白字；hover → 紫字白底；
               激活 → 白字紫底；active:hover → 显式维持白字紫底 */
.kc-subtab:not(.active):hover { border-color: var(--p-mid); color: var(--p); }
.kc-subtab.active:hover { color: #fff; background: var(--p); border-color: var(--p); }

/* kc-tab：未激活 → muted；hover → 紫字白底；
            激活 → 紫字淡紫底；active:hover → 维持紫字淡紫底 */
.kc-tab:not(.active):hover { border-color: var(--p-mid); color: var(--p); }
.kc-tab.active:hover { color: var(--p); background: var(--p-light); border-color: var(--p-mid); }
```

## Consequences

**正面**
- 所有 Tab 组件 "active + hover 紫字紫底" 死角被锁死，不会随新页面复制粘贴再次复发
- 4 处表单（含 ChangePasswordModal）用户第一次看到合理错误提示，减少盲点
- Search "输入 → 3 秒" 的等待区间有连续 loading 反馈
- RulesTab 失败态从"永远加载中"变成"加载失败 + 错误 message + 允许新建"

**负面 / 取舍**
- "去 disabled + onClick 校验" 会导致按钮始终亮着；若用户不填任何内容就点，会频繁看到红条 —— 比静默好但比即时 disabled 的 UI 反馈弱一点
- BUG-13 的 `title` 属性只在鼠标悬停可见；触屏用户看不到。若后续报告仍抱怨，要改成可视的 hint 文字
- CSS 的 `:not(.active):hover` 在旧浏览器可能不支持（IE11），但本站已使用 var() / grid 等 ES2020 feature，假设现代浏览器
