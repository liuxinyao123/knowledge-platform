# ADR 2026-04-21-14 · 真登录（G9）

## Context

G2 permissions 模型 + G3/G4 IAM UI 落地，但登录 UI / 用户表 / 密码体系 / 401 跳转全缺。
Q4=b 决策原本 park 此项，用户本次显式翻开。

## Decision

1. **零新 npm 包**：密码哈希用 Node 内置 `crypto.scrypt`，JWT 签发用 `createHmac`（与 verifyToken 对称）
2. **token 存 localStorage**：接受 XSS 自担；后续若需更高安全 → 切 httpOnly cookie（server 端 `Set-Cookie`，前端不再取 token）
3. **seed admin**：`admin@dsclaw.local / admin123`，启动时控制台警告"生产必改"
4. **DEV BYPASS 保留条件激活**：`AUTH_HS256_SECRET` 未设 + 非 prod 时兜底，避免破坏零配置开发
5. **自注册关闭**：企业内部知识中台，所有用户 ADMIN 创建
6. **Token TTL 24h，无 refresh token**：过期就重登，简单先
7. **生产 fail-fast**：`NODE_ENV=prod && !isAuthConfigured()` → `process.exit(1)`（现已实现）

## Consequences

**正面**
- 权限体系真的可以跑端到端（editor 403 / viewer 拦截都能触发）
- 生产可部署；IAM 面板连接真用户表
- 不引 bcrypt → 沙箱/CI 不受 native binding npm policy 影响

**负面 / 取舍**
- localStorage XSS 风险 —— 需要未来 CSP + httpOnly cookie 迁移
- scrypt 参数硬编码（N=16384 默认）：生产若 CPU 紧张需评估
- 无账号锁定 / rate limit → 暴力破解窗口开。短期靠 IP 层（nginx/Cloudflare）挡
- 无 refresh token → 过期即强制重登，用户体验一般
- password 明文走 HTTPS 依赖部署正确（本项目假设）
